1. Project Overview

This project visualises the real-time status of McDonald's ice cream machines across the United States, using live data from mcbroken.com — a service created by Rashiq Zahid that places a $18,752 order every minute at every U.S. McDonald's location to detect whether the ice cream machine is operational.

The goal is to transform raw machine status data into an accessible, informative, and visually compelling dashboard that communicates both the scale of the problem and the geographic distribution of broken machines.

Why this dataset? The McBroken dataset is uniquely suited to visualisation — it is real-time, geographically distributed across 13,500+ locations, and captures a phenomenon that is both humorous and genuinely useful to consumers. It also presents interesting design challenges around scale, density, and communicating probability rather than certainty.

2. Data Source & Methodology

Primary Data Source

All machine status data is sourced from data.mcbroken.com/stats.json, which provides:

  • A global broken percentage across all tracked U.S. locations
  • Per-city broken counts and rates
  • Total location counts per city

Location-level data (latitude, longitude, state, broken status) is provided via a static locations_data.js file, and state-level aggregates via state_data.js. These are pre-processed snapshots used to power the choropleth map and dot layer without requiring additional API calls.

Data Filtering

The raw dataset includes McDonald's locations outside the United States — primarily in the United Kingdom and Germany. These are filtered out client-side using a set of valid U.S. state abbreviations (US_STATES), applied to both the location dot layer and the city table. The city table additionally offers a user-controlled toggle to switch between U.S.-only and global views.

Limitations

  • Machine status is inferred from order placement, not direct hardware monitoring — a machine may be marked broken if it is temporarily offline, being cleaned, or simply slow to respond.
  • The dataset does not include historical data — all views are snapshots of current state.
  • Some international locations appear in the dataset and require client-side filtering.

3. Design Process

Visual Direction

The visual language is editorial data journalism — bold typography, high information density, and a restrained but expressive colour palette. The aesthetic draws from print data journalism (think NYT Graphics or The Pudding) adapted for an interactive web context.

Key typographic choices:

  • Bebas Neue — display headings and stat values. Chosen for its condensed boldness and strong data-display character.
  • DM Mono — labels, metadata, and UI chrome. Monospace reinforces the data-driven, technical nature of the content.
  • DM Sans — body copy and table content. Clean and readable at small sizes.

Colour Palette

The original prototype used red and green to indicate broken and working machines — a common convention that fails for the approximately 8% of users with colour vision deficiency (CVD), particularly deuteranopia and protanopia, for whom red and green are indistinguishable.

The palette was redesigned to be safe under all common CVD types:

  • Orange #9E4F00 / #F4A340 — broken/bad. Used on white and dark backgrounds respectively.
  • Teal #007A6E / #2DD4C4 — working/good.
  • Blue #0062CC / #7DC3FF — neutral/total counts.

Two variants of each colour are maintained — a dark version for use on white/light backgrounds (meeting 5:1 WCAG contrast) and a light version for dark backgrounds.

4. Accessibility

Accessibility was treated as a first-class requirement rather than an afterthought. The following standards were targeted:

  • WCAG 2.1 AA contrast ratio of 5:1 for all text elements
  • Colour is never the sole means of conveying information — icons supplement colour coding throughout
  • Full keyboard navigation support with visible focus indicators
  • Screen reader compatibility via semantic HTML, ARIA roles, and descriptive labels

Screen Reader Implementation

  • Each stat card uses role="region" and aria-label with a complete plain-English description
  • The global broken percentage uses aria-live="polite" so updates are announced
  • City table rows use aria-label summarising the full row; individual cells are aria-hidden to avoid redundant reading
  • Decorative icons are aria-hidden
  • The help modal uses role="dialog", aria-modal="true", and focus management

5. Technical Implementation

Technology Stack

D3.js v7 TopoJSON Canvas API Vanilla JS CSS Custom Properties GitHub Pages

No build tools, bundlers, or frameworks are used. The project is intentionally dependency-light — D3 and TopoJSON are loaded from CDN, and everything else is plain HTML, CSS, and JavaScript.

Map Architecture

The choropleth map uses two rendering layers:

  • SVG layer — state paths rendered via D3 and TopoJSON from the US Atlas dataset. Handles state fills, borders, hover events, and zoom transforms.
  • Canvas layer — individual location dots rendered to a <canvas> element overlaid on the SVG using CSS grid stacking. Canvas is used rather than SVG circles for performance — rendering 13,000+ SVG elements would be prohibitively slow.

The two layers are stacked using grid-template-areas, both assigned to the same grid area so they overlap perfectly without requiring explicit height on the container.

Zoom & Dot Rendering

Dots are only drawn when the zoom scale exceeds a threshold of 3× (DOT_ZOOM_THRESHOLD). At lower zoom levels the dot layer is hidden and the canvas cleared. This prevents visual noise at the national scale and focuses detail on the state level.

Dot positions are calculated by projecting geographic coordinates through D3's AlbersUSA projection, then applying the current zoom transform and a scale factor to convert from SVG viewBox coordinates to canvas pixel coordinates.

6. Iterations & Key Decisions

Initial Build

Basic choropleth with red/green colour scheme. State-level data only, no dot layer, no live API integration.

Colour Palette Overhaul

Replaced red/green with orange/teal colorblind-safe palette after accessibility review. Required separate light and dark variants for use on different backgrounds.

Canvas Dot Layer

Added individual location dots on zoom. Initial implementation used SVG circles (too slow). Switched to Canvas API with coordinate transform calculations.

Accessibility Pass

Full ARIA implementation, contrast audit to 5:1, icon supplements for colour coding, screen reader labels throughout.

International Data Filtering

Discovered dataset included UK and German locations. Added client-side US_STATES filter and user-controlled toggle on the city table.

Multi-page Navigation

Added State Rankings and Documentation pages. Navigation bar added to header across all pages.

Key Design Decisions

Considered

SVG circles for location dots — familiar D3 pattern, easy to style and interact with.

Chosen ✓

Canvas API for dots — necessary for performance at 13,000+ locations. SVG approach caused severe frame rate drops.

Considered

React or Vue for component architecture — easier state management and reactivity.

Chosen ✓

Vanilla JS — no build step required, simpler deployment to GitHub Pages, sufficient for the project's complexity level.

7. User Feedback & Iterations

The visualisation was reviewed by a peer group whose feedback informed several design changes. Feedback fell into two categories — implemented and rejected — each with documented rationale.

Implemented Feedback

Click Reminder on State Hover

Users were not discovering that states were clickable. A pill-shaped reminder now appears at the bottom of the map when hovering over a state, fading out after 2.5 seconds to avoid becoming distracting. This addresses a core discoverability problem — an interactive feature with no visible affordance is effectively invisible.

Side Tooltip on Zoom

When zoomed into a state, the cursor-following tooltip was obscuring the dot layer. A fixed side tooltip now slides in from the right of the map on hover, showing the state summary without interfering with the dot positions.

Broken Percentage Banner

The global broken percentage was previously shown only in the small header stat. Feedback indicated users wanted the headline figure to be more prominent. A dedicated banner was added between the controls bar and the map, displaying the live percentage in large type — making the most important number on the page impossible to miss.

Rejected Feedback

Requested

Show the full street address of each McDonald's location when hovering over a dot on the zoomed map.

Rejected ✕

Declined for three reasons. First, with 13,500+ locations the dot layer is already dense — address labels would cause severe visual clutter and overlapping text. Second, rendering this volume of text labels would significantly degrade performance, particularly on lower-powered devices. Third, at this scale the street address adds little analytical value — the existing tooltip already provides city, state, broken count, and rate, which is sufficient for the visualisation's purpose.

8. Future Visualisations

The following visualisations are planned for future development:

  1. State Rankings (Per Capita) — Bar chart normalised by state population, using ice cream cone iconography to represent working vs broken machines per 100k residents. Allows fair comparison between small and large states.
  2. Historical Trends — Time series of broken rates nationally and by state, pending access to historical data.
  3. Time-of-Day Heatmap — Broken rates by hour of day to identify patterns in when machines go offline.
  4. Location Search — Allow users to find their nearest McDonald's and check its current status.
